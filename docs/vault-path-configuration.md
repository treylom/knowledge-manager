# Vault Path Configuration — Technical Reference

This document is the authoritative technical reference for how knowledge-manager turns a cloned git repository into a personalised, working installation. It describes the placeholder system, the substitution engine, the skip-worktree mechanism, and the most common troubleshooting cases.

If you are a **user**, the short version is in [`README.md`](../README.md#installation--configuration).
If you are an **AI agent editing this repo**, read [`CLAUDE.md`](../CLAUDE.md) or [`AGENTS.md`](../AGENTS.md) first.

---

## Overview

knowledge-manager is distributed as a single git repository that every user clones. Skill, agent, and command files need to know where the user's Obsidian vault lives, where the Obsidian CLI executable is, and which sub-folders (Zettelkasten, Research) to use. Rather than ask every user to hand-edit dozens of files, the repo ships with **placeholder tokens** that are substituted in-place by a shell script the first time the setup wizard runs.

The design has five moving parts:

1. A **config schema** (`km-config.json`) that the wizard populates.
2. A **Node.js-backed config parser** (`scripts/_lib-config.sh`).
3. A **substitution engine** (`scripts/configure-vault-paths.sh`) that reads the config and rewrites files.
4. An **update orchestrator** (`scripts/km-update.sh`) that safely pulls upstream changes.
5. **Skip-worktree git flags** that hide local substitutions from `git status` so users don't accidentally commit their personal paths upstream.

---

## Components

| Component | Path | Role |
|---|---|---|
| Config file | `km-config.json` (gitignored) | Holds the user's vault path, Obsidian CLI path, Zettelkasten/Research roots, and other KM settings. Generated by the wizard. |
| Config template | `km-config.example.json` | Committed schema template. Users copy this if they bypass the wizard. |
| Config parser | `scripts/_lib-config.sh` | Sourced library that exposes `config_get`, `vault_name`, `path_normalize` using Node.js (no `jq` dependency). |
| Substitution engine | `scripts/configure-vault-paths.sh` | Reads `km-config.json`, walks `skills/ agents/ commands/ .claude/`, runs `sed` replacements, applies `git update-index --skip-worktree`. |
| Update orchestrator | `scripts/km-update.sh` | Unlocks skip-worktree → restores pristine HEAD → `git pull` → re-runs substitution → re-locks. |
| Setup wizard | `commands/knowledge-manager-setup.md` | Interactive wizard (Phase 4 calls the substitution engine). |
| Update command | `commands/km-update.md` | Slash-command wrapper around `km-update.sh`. |

## Placeholder Reference

| Placeholder | Config field | Default | Example value | Purpose |
|---|---|---|---|---|
| `{{VAULT_PATH}}` | `storage.obsidian.vaultPath` | *(required, no default)* | `/home/alice/Documents/MyVault` | Absolute filesystem path to the Obsidian vault root |
| `{{VAULT_NAME}}` | *derived from `basename(vaultPath)`* | *(derived)* | `MyVault` | Vault folder name — the string Obsidian CLI expects for `--vault` |
| `{{OBSIDIAN_CLI}}` | `obsidianCli.path` | `""` | `/mnt/c/Program Files/Obsidian/Obsidian.com` | Absolute path to the Obsidian CLI executable; empty string is allowed (means Obsidian CLI is unavailable) |
| `{{ZETTELKASTEN_ROOT}}` | `storage.obsidian.zettelkastenRoot` | `Zettelkasten` | `Library/Zettelkasten` | Vault-relative path to the Zettelkasten root folder |
| `{{RESEARCH_ROOT}}` | `storage.obsidian.researchRoot` | `Research` | `Library/Research` | Vault-relative path to the research/MOC root folder |

**Composition rule**: the placeholders compose trivially. For example, to reference a file under `<vault>/Zettelkasten/AI/note.md`, write:

```
{{VAULT_PATH}}/{{ZETTELKASTEN_ROOT}}/AI/note.md
```

**Windows path caveat**: on Windows, `{{VAULT_PATH}}` is normalized to forward slashes at substitution time (e.g. `C:/Users/Alice/...`). This makes the resulting paths work in both Git Bash/WSL shell commands and in most Node.js APIs. If a particular command needs backslashes, wrap the path in the shell quoting appropriate for your platform.

## Config Schema

The relevant subset of `km-config.example.json`:

```json
{
  "storage": {
    "primary": "obsidian",
    "obsidian": {
      "enabled": true,
      "vaultPath": "/path/to/your/obsidian/vault",
      "_vaultPath_examples": {
        "windows": "C:/Users/YourName/Documents/MyVault",
        "mac": "/Users/YourName/Documents/MyVault",
        "linux": "/home/yourname/Documents/MyVault"
      },
      "zettelkastenRoot": "Zettelkasten",
      "_zettelkastenRoot_help": "Vault-relative path to Zettelkasten root. Example: 'Zettelkasten' or 'Library/Zettelkasten'",
      "researchRoot": "Research",
      "_researchRoot_help": "Vault-relative path to research/MOC root. Example: 'Research' or 'Library/Research'",

      "defaultFolder": "Zettelkasten",
      "researchFolder": "Research",
      "threadsFolder": "Threads"
    }
  },

  "obsidianCli": {
    "path": "",
    "_path_help": "Absolute path to Obsidian CLI executable. Auto-detected by /knowledge-manager-setup.",
    "_path_examples": {
      "windows": "C:\\Program Files\\Obsidian\\Obsidian.com",
      "wsl": "/mnt/c/Program Files/Obsidian/Obsidian.com",
      "mac": "/Applications/Obsidian.app/Contents/MacOS/Obsidian"
    }
  }
}
```

The `zettelkastenRoot` and `researchRoot` fields are new — older configs without them fall back to the string defaults (`Zettelkasten`, `Research`). The legacy fields `defaultFolder`, `researchFolder`, `threadsFolder` are preserved for backward compatibility and are still consumed by some skills directly.

## Substitution Algorithm

`scripts/configure-vault-paths.sh` executes these 7 steps in order:

1. **Locate and validate `km-config.json`.** Exit with a clear error if the file is missing. If `storage.obsidian.vaultPath` is empty or equal to the placeholder string `/path/to/your/obsidian/vault`, exit with an error instructing the user to run `/knowledge-manager-setup`.

2. **Source `_lib-config.sh`** to expose `config_get`, `vault_name`, and `path_normalize`, then extract the 5 substitution values:
   - `VAULT_PATH` ← `config_get storage.obsidian.vaultPath` (then `path_normalize` to forward slashes)
   - `VAULT_NAME` ← `vault_name "$VAULT_PATH"`
   - `OBSIDIAN_CLI` ← `config_get obsidianCli.path ""`
   - `ZETTELKASTEN_ROOT` ← `config_get storage.obsidian.zettelkastenRoot Zettelkasten`
   - `RESEARCH_ROOT` ← `config_get storage.obsidian.researchRoot Research`

3. **Enumerate target files.** Run `find skills agents commands .claude -type f -name "*.md"`. Explicitly exclude `skills/km-storage-abstraction.md` and its `.claude/` mirror (these hold the anti-pattern example that teaches users *not* to prefix paths with the vault name).

4. **Fast-path filter.** For each file, `grep -l '{{' "$file"` to skip files that contain no placeholders. This makes the engine O(files-with-placeholders) instead of O(all-files).

5. **Apply `sed` substitutions.** For each surviving file, run `sed -i` with one substitution per placeholder, in this order (order matters because `{{VAULT_PATH}}` must expand before anything that uses its basename):
   ```
   s|{{VAULT_PATH}}|$VAULT_PATH|g
   s|{{VAULT_NAME}}|$VAULT_NAME|g
   s|{{OBSIDIAN_CLI}}|$OBSIDIAN_CLI|g
   s|{{ZETTELKASTEN_ROOT}}|$ZETTELKASTEN_ROOT|g
   s|{{RESEARCH_ROOT}}|$RESEARCH_ROOT|g
   ```

6. **Collect the list of modified, tracked files** and pass them to `git update-index --skip-worktree`. Untracked files (shouldn't happen in a clean repo, but handled gracefully) are skipped.

7. **Print a summary**: how many files were touched, which placeholders had empty expansions (warning only), and the total duration. Exit 0 on success.

## Skip-worktree Mechanism

### Why skip-worktree and not `.gitignore`?

- **`.gitignore` only works for untracked files.** Every skill file is already tracked, so `.gitignore` would not help.
- **`git update-index --assume-unchanged`** is the other option, but it is advisory — git is allowed to overwrite it on any operation. Not safe.
- **`git update-index --skip-worktree`** tells git: "this file is tracked, but the user has intentional local modifications; do not overwrite them and do not show them as dirty in `git status`." This is exactly what we want.

### What it looks like

After `configure-vault-paths.sh` runs, every substituted file has its skip-worktree bit set. Check with:

```bash
git ls-files -v skills/ agents/ commands/ .claude/ | grep '^S'
```

Each line prefixed with `S` is a skip-worktree file. `git status` will ignore modifications to it, and `git checkout HEAD -- <file>` will refuse to touch it until you unlock.

### How `/km-update` handles it

`scripts/km-update.sh` is the supported way to pull upstream changes without fighting the skip-worktree lock. Its sequence is:

1. `git ls-files -v | awk '/^S/ {print $2}'` → list of locked files.
2. `git update-index --no-skip-worktree <files>` → unlock.
3. `git checkout HEAD -- <files>` → restore the pristine placeholder form so the upcoming `git pull` sees clean files.
4. `git pull origin <current-branch>` → fetch upstream changes.
5. `bash scripts/configure-vault-paths.sh` → re-run substitution against the (possibly updated) set of files.
6. `git update-index --skip-worktree <files>` → re-lock.

If any step fails, the script aborts and prints a manual recovery hint. In practice, step 3 is the only step that can fail cleanly (if upstream changes collide with local modifications that somehow weren't placeholder-related), and the recovery is a manual `git stash` followed by re-running `/km-update`.

## Re-running Setup

Users may need to re-run the wizard if they:

- Move their Obsidian vault to a different directory.
- Rename the vault folder.
- Install Obsidian desktop and want the CLI path picked up.
- Want to switch `zettelkastenRoot` from `Zettelkasten` to `Library/Zettelkasten` (or similar reorganization).

The re-run flow:

1. User runs `/knowledge-manager-setup` again.
2. The wizard detects an existing `km-config.json` and **backs it up** to `km-config.json.bak.<YYYYMMDD-HHMMSS>`.
3. The wizard prompts for new values, seeding defaults from the existing config.
4. The wizard writes the new `km-config.json`.
5. The wizard invokes `scripts/configure-vault-paths.sh`, which automatically:
   - Unlocks skip-worktree on previously locked files.
   - Restores pristine placeholder form via `git checkout HEAD -- <files>`.
   - Re-substitutes with the new values.
   - Re-locks.

No manual cleanup is required between runs. Users can re-run the wizard as often as they like.

## Troubleshooting

### Case 1: `git status` shows every skill file as modified

**Symptom**: After running `/knowledge-manager-setup`, `git status` shows dozens of modified files under `skills/` and `agents/`.

**Cause**: The skip-worktree flag was never applied (or was cleared by a git operation such as `git reset --hard`).

**Fix**:

```bash
# Re-apply skip-worktree by running the substitution engine
bash scripts/configure-vault-paths.sh

# Verify
git ls-files -v skills/ | grep '^S' | wc -l
# Expected: a non-zero number matching the count of substituted files
```

If the problem persists, check that `scripts/configure-vault-paths.sh` is executable (`chmod +x scripts/configure-vault-paths.sh`) and that your `km-config.json` has a valid `storage.obsidian.vaultPath`.

---

### Case 2: `git pull` fails with "your local changes would be overwritten"

**Symptom**:

```
error: Your local changes to the following files would be overwritten by merge:
    skills/km-workflow.md
Please commit your changes or stash them before you merge.
```

**Cause**: You ran `git pull` directly instead of `/km-update`. Upstream changed a line inside a file that has your substituted values, and git refuses to merge because the working-tree copy no longer matches HEAD.

**Fix**:

```bash
# Preferred: use the supported update path
/km-update

# Or, if you need to do it manually:
git ls-files -v | awk '/^S/ {print $2}' > /tmp/skipped.txt
xargs -a /tmp/skipped.txt git update-index --no-skip-worktree
xargs -a /tmp/skipped.txt git checkout HEAD --
git pull
bash scripts/configure-vault-paths.sh
```

**Prevention**: always use `/km-update` instead of raw `git pull` once you have run the setup wizard.

---

### Case 3: A placeholder wasn't substituted — skill file still contains `{{VAULT_PATH}}`

**Symptom**: You run a skill and see a literal `{{VAULT_PATH}}` in an error message or log line.

**Cause**: One of:
- The file was added after the last `configure-vault-paths.sh` run (e.g. you pulled upstream without going through `/km-update`).
- Your `km-config.json` is missing the relevant field and the default fell through as an empty string.
- The file path was excluded by the explicit `km-storage-abstraction.md` carve-out (expected — that file is an anti-pattern teaching example).

**Fix**:

```bash
# 1. Re-run the substitution engine to cover newly-added files
bash scripts/configure-vault-paths.sh

# 2. If the issue persists, check your config
cat km-config.json | node -e "const c=require('./km-config.json'); console.log('vault:', c.storage?.obsidian?.vaultPath); console.log('cli:', c.obsidianCli?.path)"

# 3. Verify the file is not the anti-pattern exclusion
grep -l '{{' skills/ .claude/skills/ -r
```

If `vault` prints an empty string or `/path/to/your/obsidian/vault`, re-run `/knowledge-manager-setup` and fill in a real path.

---

### Case 4: Obsidian CLI path is empty and skills fail with "command not found"

**Symptom**: Skills that call Obsidian CLI (e.g. `"" create --vault ...`) fail because the CLI path was substituted to an empty string.

**Cause**: The user does not have Obsidian desktop installed, or the wizard's auto-detection could not find the binary. `{{OBSIDIAN_CLI}}` legitimately substitutes to `""` in this case because the field is optional.

**Fix**:

1. Install Obsidian desktop from <https://obsidian.md/>.
2. Verify the CLI is present:
   ```bash
   # WSL
   ls "/mnt/c/Program Files/Obsidian/Obsidian.com"
   # macOS
   ls "/Applications/Obsidian.app/Contents/MacOS/Obsidian"
   ```
3. Edit `km-config.json` directly, set `obsidianCli.path` to the correct absolute path, then:
   ```bash
   bash scripts/configure-vault-paths.sh
   ```
4. Alternatively, re-run `/knowledge-manager-setup` and let the wizard auto-detect.

If you do not want to use Obsidian desktop at all, that is fine — knowledge-manager has a 3-Tier fallback (CLI → MCP → direct file write) and skills will automatically fall back to the next tier when `{{OBSIDIAN_CLI}}` is empty. Look at `skills/km-environment-detection.md` for details.

---

### Case 5: Mirror drift — a skill file under `skills/` is out of sync with its `.claude/` twin

**Symptom**: `diff -q skills/foo.md .claude/skills/foo.md` reports the files differ, or a verification CI job rejects a PR because mirrors are out of sync.

**Cause**: Someone edited one copy without updating the other. Common after a merge or a hand-crafted commit that touched only the top-level path.

**Fix**:

```bash
# Option A: keep the skills/ version as source of truth
cp skills/foo.md .claude/skills/foo.md

# Option B: keep the .claude/ version
cp .claude/skills/foo.md skills/foo.md

# Then re-run substitution (the edit may have introduced a new placeholder)
bash scripts/configure-vault-paths.sh

# Verify no drift remains
for f in skills/*.md agents/*.md commands/*.md; do
  diff -q "$f" ".claude/$f" || echo "STILL OUT OF SYNC: $f"
done
```

**Prevention**: always edit both mirrors in the same commit. `CLAUDE.md` and `AGENTS.md` document this invariant and provide the relevant check commands.
